This lays out requirements for the front end interface.
It is written in pure HTML/CSS/JS.

There are 3 pages.
Homepage: "/"
Login page: "/login"
Game page: "/game/<game id>"
User info page: "/user/<user id>"
Change login info page: "/changeinfo"

Homepage (home.html):

The homepage has:
 - a header bar with the game name on the left (HexToe)
 - either login button or username + logoout button on the top right in the header bar
 - Two main buttons; "Play" and "Create Private Room"
 - a list of recently active games
 - a live view of the selected game from the list (default to most recent)

Homepage functionality:
 - When the user loads the page, js should immediately try to get current user info via the "session_id" http only cookie. If it succeeds, set a variable `userinfo` to the result of that call. If not, set `userinfo` to NULL.
 - Whenever we set `userinfo` (via a setter), update the login/logout interface in the top right. There should be a login button if `userinfo` is NULL, and the user's username + a logout button otherwise.
 - The login button should send the user to the "/login" page.
 - The logout button should simply call the logout endpoint and set `userinfo` to NULL
 - When the user clicks "Play", js should call the join_game endpoint. This returns a game id, and the user should automatically be navigated to "/game/<game_id>"
 - When the user clicks "Create Game", js should call the create_private_game, which returns a game id, and once again automatically navigate to "/game/<game_ids>"
 - The active games are displayed, as well as a live view of an active game. This is loaded when the page is loaded.
 - By default, the most recent active game is displayed. The user may mouse over any game entry in the active game list, which will change the display to show that game
 - The user may click on any game entry in the list, which will send them to the game page with that game's game_id
 - The user can pan around in the game viewer, but not make any moves


Login page (login.html):

The login page has:
 - a header bar with the game name on the left
 - a username text box
 - a password text box
 - "Login" button below
 - "Create Account" button below as well

Login page functionality:
 - When the user clicks login, it should send that username and password to the login endpoint, and hopefully set the session_id cookie when it returns from the endpoint. If it gets 403, that information must be reflected to the user. The same applies to create account, except that calls the endpoint for create_user.
 - Whenever we successfully login or create an account, and the sessiond_id has been set, we navigate back to the homepage.


Game page (game.html):

The game page has:
- Header with game title and login/logout button in top right, same as homepage
- A large HTML canvas for the main game area
- a scrollable list of move history on the right hand side
- text above the move history indicating if it is the current user's turn
- A button above the move history prompting the user to join the match

Game page functionality:
 - The game state should be stored in `game_state`. Whenever `game_state` is set, clear and redraw the canvas based on its content. To draw it, draw a hexagonal grid over the entire canvas. Then, populate the hexagonal grid with Xs and Os based on the moves list. Any hexagonal tile indicated with p1 == true should have an x at the tile at <a, r, c>, whereas p1 == false should have an O at that tile.
 - Every 5 seconds, request the game state from the server via `get_game`. Update `game_state` via the setter, which will update the canvas.
 - If the game state says is_your_turn is true, and the player clicks on a tile, send a make_move request to the server with that tile's location. The response will be the new game state, which will update the canvas.
 - Above the move history, there can be either text or a button. If the current user is already in this game, it displays:
    - A "Join Game" button if there is only 1 player ID in this game AND it's not the current user. This should trigger the join_specific_game endpoint. Otherwise...
	- Text saying "Waiting for opponent..." if there is only 1 player id in this game. Otherwise...
	- Text saying "Your Turn" if is_your_turn is true, "Opponent's Turn" if is_your_turn is false


User info page (user.html):

The user info page has:
 - The user's username
 - A list of games they've played
 - If the user is looking at their own page, they will have a button to change their login info.
	This button should send them to "/changeinfo"
 - The list of games they've played will look the same as the game list from the homepage.
 - Like the homepage, the user can click on a game in the list.
 - Unlike the homepage, there will not be a canvas showing the currently selected game


Change info page (change_info.html):

The change info page has:
 - A text box (Username) auto filled with the user's current username
 - A password box (Password) which is empty to start
 - A text box (New Username) auto filled with the user's current username
 - A password box (New Password) which is empty to start
 - Info text below: "Leave the New Password box blank to keep your current password"
 - "Submit" button

This page should redirect to the login path if the user does not have a session token, aka their user info returns null.
When the user clicks submit, give an error if either username box is empty, or if the password box is empty
Otherwise, when they click submit, first call login with the Username and Password. If this fails, give the user an error (Incorrect login info)
Otherwise, populate a dict with the new username, and the new password if it is not empty.
Submit this dict to the update_user endpoint.
If successful, clear the text boxes and show a green message saying "Updated your login info"
